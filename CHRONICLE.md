### Phase 8: Virtual Scrolling & Streaming Data Loading (2026-05-28)
- **Virtual Scrolling:** Replaced full-DOM tab rendering with window-scroll virtual lists using `@tanstack/virtual-core@3` (CDN). Off-screen rows are unmounted, allowing large datasets to render smoothly without memory pressure.
- **SAX Streaming JSON Parse:** Introduced `@streamparser/json` (CDN) for Phase 1 parallel streaming. Each of the 13 PRIMARY data files streams in independently; batches of 30 items (or every 50ms) trigger incremental UI updates so data appears progressively as it downloads.
- **Two-Phase Data Loading:** Separated data keys into `PRIMARY_DATA_KEYS` (13 display-critical files, streamed) and `ENRICHMENT_DATA_KEYS` (8 enrichment files, loaded normally). Phase 1 gives users immediately usable data; Phase 2 enriches it silently in the background.
- **Tab Descriptor Pattern:** Refactored `TabRenderers` to return `{ items, renderItem, estimateSize }` descriptors instead of HTML strings, decoupling list data from rendering infrastructure.
- **Smart Re-render Detection:** `TabRendererRegistry` skips re-renders when `DataRepository` reference and filter fingerprint are both unchanged, preventing redundant paints during streaming.
- **Bug Fixes — extendItems Safety:** Discovered that `extendItems` (which reuses cached row DOM by index) is unsafe for `.reverse()` arrays because each streaming batch reshuffles every existing index. Also fixed a case where clearing a search filter (increasing item count) would incorrectly take the `extendItems` path. Fixed by removing `extendItems` from `TabRendererRegistry` entirely — all updates now go through `refreshItems`.
- **Bug Fixes — loadedDataKeys Consistency:** Fixed `#streamKey` fetch/HTTP error paths that left keys absent from `loadedDataKeys` indefinitely. Fixed `load()` not awaiting `#loadEnrichment()`, which caused the CI validator to fail on `metadata`.

### Phase 14: Initial-Tab Priority Loading (2026-06-13)
- **Phase 0 in `DataLoader.load()`:** The default tab is 公式 (official), but its data (`officialPages` 14KB + `officialNews` 1.3KB) used to load in Phase 2 (ENRICHMENT) — after all 13 PRIMARY files (several MB) finished streaming. A new Phase 0 now loads `INITIAL_TAB_DATA_KEYS = ["officialPages", "officialNews"]` first and alone, so the initial screen renders after ~16KB instead of after megabytes.
- **Plain fetch over streaming for Phase 0:** Streaming would have to await the `@streamparser/json` CDN import; for 16KB, `loadKeys()` (fetch + `JSON.parse`) is strictly faster. The CDN import is still fired (un-awaited) at Phase 0 start so it warms in parallel and Phase 1 is not delayed.
- **No double-load:** the two keys were removed from `ENRICHMENT_DATA_KEYS`; `CORE_DATA_KEYS` now includes `INITIAL_TAB_DATA_KEYS` so the Node validator's `load()` coverage assert still covers them.
- **Adversarial-Review Fixes (same day):** A two-agent refutation workflow on the change found real regressions, all fixed: `fetchJsonText` silently swallowing HTTP errors (official tab would render blank with no error card) → now throws, with per-key fallback + `dataLoadErrors` recording in `loadKeys()` matching `#streamKey` semantics; the un-awaited `getJSONParser()` warm-up could fire `unhandledrejection` during the Phase 0 window → `.catch(() => {})`. Pre-existing issues fixed alongside: a parser-CDN outage rejected `load()` and wiped the whole viewer → `#streamKey` now degrades to plain fetch+parse via `getJSONParser().catch(() => null)`; the blocking static `lite-youtube-embed` import gated the entire boot on esm.sh → un-awaited dynamic import (custom elements upgrade on late definition). Streaming flushes re-rendered the now-visible official tab every ~200ms (collapsing open `<details>`) → new optional descriptor field `itemsSource` lets `tab-registry` skip refreshes while the source arrays are reference-identical. The Node validator's vacuous `load()` coverage assert for streamed keys was recorded in `NEXT_TASKS.md` (P2).

### Phase 15: Phase 1 Streaming Render-Cost Reduction (2026-06-13)
- **renderSummary no longer builds rich collections:** A benchmarked multi-agent investigation found `invalidate()` is effectively free (~0.5µs) but `ViewController.renderSummary()` ran on every patch and unconditionally read `repo.richBouts/richFighters/richVideos`, rebuilding all three rich collections on every one of ~37–180 Phase-1 flushes (~4ms each = ~150–730ms wasted on partial, soon-discarded data — even on the official tab whose content render is already skipped). Fix: 試合/動画 counts now use the plain `d.bouts.length`/`d.videos.length` (rich is reverse + stable partition only, so the count is provably identical); 選手 uses `repo.richFighters.length` only once its real dependencies (`fighters`+`numbersFighters`+`numbersNameMatches`+`officialPlayers`) are loaded, else the plain `fighters.length`. Final displayed counts are unchanged; the gate is also more correct (the old intermediate richFighters count during Phase 1 was wrong, since numbers/official discovery hadn't loaded).
- **Patch coalescing:** `#streamKey` flush now calls `#schedulePatch()` (one `requestAnimationFrame` per frame, shared across the 13 parallel keys) instead of `state.patch({})` per flush; `invalidate()` stays per-flush. Node/tests (no rAF) fall back to immediate patch. Per-phase confirm patches stay direct, guaranteeing a settled final render.
- **renderViewModeSwitch guard:** skips the innerHTML rebuild when `viewMode` is unchanged and already painted.
- **Trimmed unused PRIMARY keys:** removed `aliases` (an object mis-streamed through the array SAX path to `[]`, and unread — fighter aliases live in fighters.json records) and `titleReigns` (base getter only, no caller; lineage is baked into titles.json) from `PRIMARY_DATA_KEYS` (13→11 streamed keys). Files remain in `DATA_FILES`. A parallel `metadata` mis-parse (object in ENRICHMENT) and three un-memoized per-render filters were recorded as follow-ups in NEXT_TASKS.

### Phase 16: Phase 2 Enrichment/Reference Load Optimization (2026-06-13)
- **Un-gated public references:** A consumer-mapping investigation found `load()` did `await #loadEnrichment()` *then* `await loadPublicReferences()`, so the ~1.5MB of `source*References` shown on public bout/event/video cards (and woven into public search) did not start until all enrichment — including the ~1MB admin-only `source_mentions` — finished. Replaced with a single `Promise.all([#loadEnrichment(), loadPublicReferences()])`; references are independent of the enrichers, so parallelizing is behavior-preserving and makes public Phase-2 data ready ~57% sooner (6-connection FIFO model). Both helpers use `#streamKeySafe`, so the combined pool can't reject `load()`.
- **Deferred `source_mentions` (largest Phase-2 file, ~1MB) to its admin tab:** every reader was enumerated — `richSourceMentions` (admin 出典言及 tab only) and `countSourceMentions` (one public fallback badge). Removed it from `ENRICHMENT_DATA_KEYS`; it stays in `TAB_DATA_KEYS.mentions` + `REQUIRED_TAB_DATA_KEYS.mentions` (lazy load on tab open, covered by the existing loading-gate card). The public fallback badge (mention counts on a video lacking a source reference) is now gated on `loadedDataKeys.has("sourceMentions")` so it shows nothing rather than misleading zeros until loaded. Public-only users never download the 1MB.
- **Dropped dead `metadata`** from `ENRICHMENT_DATA_KEYS` (zero readers; object mis-streamed to `[]` — same class as the earlier aliases/titleReigns removal).
- **Kept eager (proven public-critical):** `source_documents` (tsushin tab + video/article detail panels), the three `source*References` (public cards + search), `official_tournaments` (public events badge). `official_matches`/`official_history` deferral was left as an optional NEXT_TASKS follow-up (admin-only but tiny, and needs officialMisc wired into the on-demand maps).

### Phase 17: Tsushin Tab Un-gated From Body Load (2026-06-13)
- **Progressive render for the 通信 (tsushin) public tab:** A post-load runtime investigation found tsushin was hard-gated on the full ~557KB `sourceDocumentBodies` download — `REQUIRED_TAB_DATA_KEYS.tsushin` listed both `sourceDocuments` and `sourceDocumentBodies`, so opening the tab showed only a "読み込み待ち" placeholder (rendering zero of the 120-card list) for the whole download, even though `source_documents.json` (eager Phase 2) already carries a `content_preview` for every note doc and `renderNoteArticleCard` falls back to the preview when `content_text` is absent. Changed the required keys to just `["sourceDocuments"]`; `TAB_DATA_KEYS.tsushin` still streams the bodies on tab open, so the list renders instantly from previews and each card upgrades to its full inline note body as bodies arrive (each flush bumps `repo.revision` → tab-registry re-render). Behavior-preserving final state; turns a multi-hundred-KB blank wait into an instant, progressively-enriched list.
- **Confirmed already-optimal (left unchanged per the investigation's doNot):** search stays warm via `QueryMatcher.#cachedText` (revision-keyed) and keystrokes never invalidate; tab-switch/focus gating is correct; extending the `itemsSource` skip to data tabs would be a no-op (their items change with filters/search); the three un-memoized filters are below the noise floor at this data scale. A note-only bodies file split and the filter memoization were recorded as optional NEXT_TASKS follow-ups.

### Phase 18: First-Paint Critical Path (CDN / Fonts) (2026-06-13)
- **Dynamic `@tanstack/virtual-core` import:** An investigation of the non-data-load performance surface (service worker, first-paint, payload) found the one real remaining win: `virtual-list.js` statically imported `@tanstack/virtual-core` from esm.sh, so the boot module graph (`main.js` → `tab-registry` → `virtual-list`) blocked on a cold third-party round trip before `main.js` top-level — including `dataLoader.load()` (Phase 0) — could run. (The team had already made `lite-youtube-embed` dynamic for exactly this reason; virtual-core slipped through.) `Virtualizer`/`windowScroll` are used only in `#instantiateVirtualizer` at render time, never at construct time, so the import was made dynamic + prewarmed at module eval (overlapping Phase 0's data fetch). A `#virtualizerGen` guard prevents double-instantiation if `setItems` is called before the module resolves; a `.catch` (resetting the cached promise) avoids an unhandled rejection and allows retry. esm.sh-down now degrades to a "読み込み中…" list instead of failing the whole module graph (a boot-blocking improvement, not just a perf one).
- **Resource hints + un-chained fonts:** Added `preconnect` for esm.sh / fonts.googleapis.com / fonts.gstatic.com, and moved the Google Fonts `@import` out of `style.css` (a render-blocking request chained behind the stylesheet) into an `index.html` `<link>` discovered at HTML-parse time. `display=swap` + system-ui fallback unchanged.
- **Explicitly rejected (doNot, measured):** minifying generated JSON (gzip already absorbs the indent — 4.5–6.7% savings — and it breaks stable diffs); SW app-shell precaching (no deploy-time cache-busting → would serve stale JS forever; HTTP cache already makes repeat loads fast); PWA manifest; critical-CSS inlining; header.webp preload; modulepreload — all below the noise floor or not worth the tradeoff.

## Key Architectural Decisions
- **Static First:** Chose a static site architecture (CSV -> JSON -> GitHub Pages) for low maintenance and high availability.
- **Human-in-the-loop:** Decided to use `review/` CSVs for all automated extractions to ensure high data quality.
- **Context-Rich Viewer:** Focused on showing source context directly in the viewer to aid data verification.
- **Clean Build, Rich Client:** Decided to keep the build process (CSV to JSON) strictly factual and perform all data supplementation, cross-referencing, and Numbers-data merging on the client side at runtime.
- **Numbers Data as Absolute Truth:** Decided to treat human-curated Apple Numbers data as the absolute source of truth for the viewer, allowing it to unconditionally overwrite canonical CSV data when matches are confirmed.
- **Archives as External Metadata:** Treats archive CSVs as committed source metadata for display and review. Archive rows can enrich labels/search but do not confirm winners, fighter identities, methods, or title lineage.
- **Service Worker as Transparent Cache Layer:** GitHub Pages does not support custom `Cache-Control` headers, so a Service Worker was chosen to implement stale-while-revalidate for JSON data files. Clients always get a fast cached response; the SW revalidates in the background and notifies clients when data changes without blocking the page.
- **CDN Libraries with Quality Gate:** CDN dependencies are adopted only when actively maintained (recent release within ~12 months) and addressing a genuinely complex problem. Preference for high-star-count, widely-used libraries. Rejected mark.js (unmaintained since 2022). Surveyed and confirmed no better alternatives for the three adopted libraries.
- **Official Data Enrichment is Client-Side Only:** Official site data from `kobayashi856/arakaku-site` is integrated as a second enrichment layer (alongside Numbers data) entirely in `data-enricher.js`. `build_json.py` is never modified for enrichment — it outputs canonical CSV facts only. This keeps the Python build pipeline decoupled from the client data model.
- **Keyboard Navigation as Core Feature:** Added vim-style keyboard navigation (j/k cursor, h/l tabs, Enter activate, etc.) with a virtual list cursor and a `?` help dialog. URL permalink sync (`url-sync.js`) allows state sharing and deep linking without server-side routing.
