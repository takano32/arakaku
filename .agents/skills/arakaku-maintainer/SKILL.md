# ARAKAKU Maintainer Skill

This skill is for working on the `takano32/arakaku` repository.

ARAKAKU is a CSV-backed static database and GitHub Pages viewer for organizing ARAKAKU promotions, events, bouts, fighters, titles, videos, source documents, and extracted source mentions.

Use this skill whenever the task touches:

- `data-src/*.csv`
- `review/*.csv`
- `scripts/*.py`
- `tests/*.py`
- `docs/index.html`
- `docs/assets/js/*.js`
- `docs/assets/style.css`
- `.github/workflows/*.yml`
- `Makefile`
- `README.md`
- `AGENTS.md`
- `SCHEMA_NOTES.md`

---

## Project model

The project has three layers.

1. Source CSVs

   Canonical data lives in `data-src/*.csv`.

2. Generated JSON

   `scripts/build_json.py` converts CSVs into `docs/data/*.json`.

3. GitHub Pages viewer

   `docs/index.html`, `docs/assets/js/`, and `docs/assets/style.css` render the generated JSON.

Do not edit generated JSON directly. Always edit source CSVs or scripts, then rebuild.

## Critical rules

### Do not edit generated JSON directly

Never directly edit:

```text
docs/data/*.json
```

These files are generated by:

```bash
make build
```

or:

```bash
make check
```

After local checks, generated JSON is normally removed with:

```bash
make clean-generated
```

### Do not commit local cache files

Do not commit:

```text
tmp/note-html/*.html
tmp/youtube-info/*.info.json
__pycache__/
.pytest_cache/
```

Only `.gitkeep` files under `tmp/` should be tracked.

### Do not invent confirmed facts

Do not infer or invent:

- bout winner
- bout loser
- method
- round
- time
- fighter identity
- event identity
- title lineage

If the source is uncertain, keep it as a candidate in `review/`, or mark it as unknown.

Use these patterns when data is unclear:

```text
result_status=unknown
勝敗未入力
needs_review
review/*.csv candidate
```

### Review before large data changes

Automatically extracted data must first go into `review/`.

Do not bulk-apply extracted results directly to `data-src/bouts.csv`, `data-src/fighters.csv`, or other canonical CSVs without creating review candidates and validating them.

## Canonical data files

### `data-src/promotions.csv`

Promotion and organization data.

Examples:

- ターゲット
- エンペラー
- マウンテン・ヒーローズ
- MAXバウト
- エリートスピリッツ

Use this for promotion names, categories, rules, and summaries.

### `data-src/events.csv`

Event data.

`event_id` is a key used by:

- `bouts.csv`
- `video_links.csv`
- source mention matching
- viewer event navigation

### `data-src/bouts.csv`

Bout data.

Important fields and meanings:

- bout-level facts live here
- `result_status`: use `unknown` when not confirmed
- method / round / time fields: only fill when confirmed

The build process remains strictly factual. The viewer automatically supplements `unknown` results, divisions, and formats using Numbers data at runtime and displays a "名鑑確認済み" badge.

Do not derive winner/loser from matchup ordering.

### `data-src/bout_participants.csv`

Bout participant relationship data.

Important fields and meanings:

- `bout_id`: links to `bouts.csv`
- `side`: participant side, currently `red` or `blue`
- `fighter_id`: links to `fighters.csv` when the fighter is identified
- `fighter_name`: source/display name preserved even if identity is uncertain
- `result`: `win`, `loss`, or `unknown`

Viewer `matchup`, `winner`, and `loser` fields are derived from this table during JSON generation.

### `data-src/fighters.csv`

Fighter data.

Use for fighter IDs, display names, divisions, gyms, aliases, and summaries.

### `data-src/numbers_fighters.csv`

Numbers-derived fighter profile rows from `data-raw/アラカク選手名鑑.numbers`.

Important fields and meanings:

- `numbers_fighter_id`: primary key for the Numbers row
- `source_row`: row number in the Numbers sheet
- `display_name`: fighter name as written in Numbers
- `fight_count`, `wins`, `losses`, `win_rate`: Numbers aggregate stats
- `white_glove_count`, `tournament_win_marker`, `tournament_entry_raw`, `belt_marker`: Numbers achievement columns

This table does not replace `fighters.csv`.

### `data-src/numbers_name_matches.csv`

Inferred mapping from Numbers fighter rows to canonical fighters.

Important fields and meanings:

- `numbers_fighter_id`: links to `numbers_fighters.csv`
- `matched_fighter_id`: exact-match canonical `fighters.fighter_id`, when found
- `candidate_fighter_id`: existing match or generated candidate ID
- `match_method` and `match_confidence`: make the matching rule explicit

Treat these mappings as review aids, not confirmed identity facts when the match is only a generated candidate.

### `data-src/numbers_fight_records.csv`

Personal fight-record rows from the Numbers "個人成績" sheet.

Important fields and meanings:

- one row is one fighter's view of one fight
- the same bout may appear once for each fighter
- some rows may be one-sided
- contradictory result marks must be surfaced in the viewer, not silently resolved

The viewer uses this table to supplement `unknown` canonical bout results via `DataRepository.getRichBoutInfo` at runtime. Do not directly convert this table into `bouts.csv` or `bout_participants.csv`.

### `data-src/titles.csv`

Title and tournament entity data.

Use for belt, title, and tournament entities.

### `data-src/title_reigns.csv`

Title and tournament lineage data.

Use for reign order, fighter, source article/video, and won/lost event references.

### `data-src/fighter_snapshots.csv`

Time- or event-specific fighter state.

Use for historical attributes such as division, gym, title status, or profile at a given event.

### `data-src/videos.csv`

Video metadata.

Videos are independent entities. Do not embed videos directly into bouts or events.

### `data-src/video_links.csv`

Links videos to entities.

Supported `entity_type` values:

```text
event
bout
fighter
promotion
title
```

Expected `relation_type` values include:

```text
full_fight
highlight
short
stream_archive
preview
interview
commentary
reference
```

### `data-src/article_links.csv`

Links articles to entities.

Supported `entity_type` values include:

```text
event
bout
fighter
fighter_snapshot
promotion
title
title_reign
video
```

### `data-src/articles.csv`

Source article metadata, especially note articles.

Deleted, private, or 404 note articles may remain in this CSV if they are historically useful. Missing cache does not necessarily mean the article row should be removed.

### `data-src/source_documents.csv`

Full text source database.

Contains:

- note article body text
- YouTube description text

Common `source_type` values:

```text
note_article
youtube_description
```

### `data-src/source_mentions.csv`

Extracted mention candidates from source documents.

Common `mention_type` values:

```text
event
matchup
result
note_url
youtube_url
```

This file supports review and future result extraction. Treat entries as candidates unless separately confirmed.

### `data-src/archives/youtube.csv`

Archived YouTube metadata generated from `tmp/youtube-info/*.info.json`.

Important fields:

- `display_id`: primary key, usually the YouTube video ID
- `webpage_url`: source URL
- `fulltitle`: archived title
- `uploader`: archived channel/uploader
- `upload_date`: archived upload date from yt-dlp
- `description`: archived description text
- `archived_at`: first archive time, preserved on regeneration for existing rows

Use this for viewer display/search enrichment and metadata review. Do not use it to confirm bout results, fighter identity, or title lineage.

### `data-src/archives/note.csv`

Archived note metadata generated from `tmp/note-html/*.html`.

Important fields:

- `filename`: primary key, cache filename
- `webpage_url`: canonical note URL
- `title`: archived HTML title
- `description`: archived meta description
- `archived_at`: first archive time, preserved on regeneration for existing rows

Use this for viewer display/search enrichment and metadata review. Do not use it to confirm facts without checking the source context.

### `data-src/aliases.csv`

Alias and spelling variation data.

Use this to improve search, matching, and entity resolution.

`review/` contains candidate data before it becomes canonical.

Examples:

```text
review/note_result_candidates.csv
review/note_structured_results.csv
review/structured_result_patch_candidates.csv
review/youtube_description_candidates.csv
review/inferred_bouts_from_video_titles.csv
review/inferred_events_from_video_titles.csv
review/inferred_fighters_from_video_titles.csv
review/source_mention_result_candidates.csv
review/source_event_reference_candidates.csv
review/source_bout_reference_candidates.csv
review/source_video_reference_candidates.csv
review/parse_skips.csv
```

When building new extraction workflows:

1. Write candidates to `review/`.
2. Inspect candidate counts and confidence.
3. Confirm mappings.
4. Apply only reviewed candidates to `data-src/`.
5. Run `make check`.
6. Run `make clean-generated`.

Builds `docs/data/*.json` from `data-src/*.csv`.

Expected outputs include:

```text

promotions.json
events.json

youtube_archives.json
note_archives.json
source_event_references.json
source_bout_references.json
source_video_references.json
```

### `scripts/validate_json.py`

Validates generated JSON.

Checks include:

- duplicate IDs
- missing required fields
- unknown references
- invalid video links
- duplicate source IDs
- duplicate mention IDs

### `scripts/cache_note_html.py`

Caches note article HTML to:

```text
tmp/note-html/
```

404, deleted, or private articles should not stop the whole pipeline.

### `scripts/cache_youtube_info.py`

Caches YouTube `.info.json` via `yt-dlp` to:

```text
tmp/youtube-info/
```

Do not download video bodies.

Warnings from `yt-dlp` are not always fatal if `.info.json` is generated and descriptions are present.

### `scripts/build_source_documents.py`

Builds:

```text
data-src/source_documents.csv
data-src/source_mentions.csv
```

from:

```text
tmp/note-html/
tmp/youtube-info/
```

### `scripts/archive_metadata.py`

Builds:

```text
data-src/archives/youtube.csv
data-src/archives/note.csv
```

from:

```text
tmp/youtube-info/*.info.json
tmp/note-html/*.html
```

The script uses stable headers and stable sort order, and preserves existing `archived_at` values for existing archive rows.

### Result extraction scripts

The following scripts are for extraction and review workflows:

```text
scripts/extract_note_result_candidates.py
scripts/extract_note_structured_results.py
scripts/make_source_mention_result_candidates.py
scripts/make_source_reference_candidates.py
scripts/make_structured_result_patch_candidates.py
scripts/apply_structured_result_patches.py
```

Do not apply result patches without reviewing candidate CSVs.

## Viewer rules

Viewer files:

```text
docs/index.html
docs/assets/js/main.js
docs/assets/js/config.js
docs/assets/js/core/
docs/assets/js/ui/
docs/assets/js/services/

```

Current viewer tabs:

```text
試合
選手
大会
団体
王座
動画
出典本文
出典言及
```

Current viewer behavior also includes:

- related source candidates on bout, event, and video cards
- YouTube description previews on video cards
- inline `▶ 詳細` / `▼ 詳細` disclosure controls for note article links, source candidate note links, and video links
- candidate labels for inferred or extracted source references

Expected navigation behavior:

- Fighter links open the fighter view and focus that fighter.
- Event links open the event view and focus that event.
- Event cards show related bout cards.
- Fighter cards show related bouts.
- Source document view shows note bodies and YouTube descriptions.
- Source mention view shows extracted event, matchup, result, note URL, and YouTube URL candidates.

After viewer changes, run:

```bash
make check
make clean-generated
```

Then verify Pages after pushing:

```text
https://takano32.github.io/arakaku/
```

## Large-change checklist

Before a large data change:

- Confirm source files.
- Confirm candidate CSVs.
- Check duplicate IDs.
- Check unknown references.
- Check that no generated JSON is committed.
- Run `make check`.
- Run `make clean-generated`.

Before applying many result candidates:

- Confirm event IDs.
- Confirm fighter IDs.
- Confirm that winner/loser are explicitly supported by sources.
- Confirm method/round/time from source text.
- Check for same-name or similar-name false positives.
- Keep ambiguous rows in `review/`.

Before changing viewer behavior:

- Verify tab rendering.
- Verify search.
- Verify fighter links.
- Verify event links.
- Verify source document and source mention tabs.
- Run `make check`.
- Check Pages after push.
